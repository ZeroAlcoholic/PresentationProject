數位與數據組織整併藍圖

本報告以「不縮減科組、但減少部門數」為前提，重整數位發展部、數據暨人工智慧發展部、行銷資訊部、中台資訊部的單位職掌，依能力邏輯、流程責任、治理機制與組織跨度，提出三個可執行方案與一條分階段推進路徑。

核心結論：先拆能力邊界，再整部門版圖 主體範圍：數位、數據、前台交付、中台工程 不納入第一波：雲端、策略、系統、安全等平台治理部門 設計原則：邊界清楚、主管幅度合理、治理責任完整
總結

先看結論

建議方向

方案 A 最完整。前台交付、智慧分析與工程中台各自歸位，責任鏈最清楚。

關鍵判斷

數位與數據應協作，不宜直接合成單一部門，否則分析與治理會被前台需求擠壓。

討論範圍

主體聚焦數位、數據、前台交付與中台工程；其他部門列為接口或下一圈議題。

判斷基準

組織設計四原則

  • 資料治理、數據工程與智慧分析，必須保有獨立投資與治理空間。

流程剖析

End-to-End 流程

1. 通路與客戶需求形成 數位商務發展科、數位客戶經營科、數位支援科、業務平台科與需求側通路部門定義場景、產品、會員與通路需求。
2. 前台系統分析與交付 電子商務資訊科、客戶服務資訊科、通路支援資訊科承接前台系統開發、維運與行銷科技落地。
3. 資料整備與平台供應 資料倉儲科、數據架構科、數據工程組負責資料導入、平台建置、模型服務與資料環境穩定供應。
4. 智慧分析與治理 客戶智能科、商業智能科、資料治理科建立客戶視圖、分析模型、資料標準與品質治理，支撐跨部門決策。
5. 核心服務與共用技術 業務中台科、元件組裝科、技術中台科、桃園軟體開發科，以及依方案調整的台中軟體開發科，提供壽險核心需求、共用元件與技術平台，支撐前台與數據能力長期擴充。

關鍵斷點在第二步與第三步之間，以及第四步與第五步之間。若組織設計沒有把前台交付、資料平台、智慧分析、中台工程分清楚，整併後會持續出現需求優先順序衝突。

正式方案

三個可成立的部門整併方案

三個方案都能把部門數由四降三，但差異不在於「降幾個部門」，而在於前台交付、資料平台與中台工程分別放在哪一條責任鏈，以及整併後的治理成本由誰承擔。

方案 A 主案 平衡型

重組前台交付與資料平台，讓前台、數據、中台各自歸位

部門轉換：`數位發展部 + 行銷資訊部 + 數據暨人工智慧發展部 + 中台資訊部` → `數位發展部（擴編） + 數據暨人工智慧發展部（擴編） + 中台資訊部（擴編）`

此方案將四個部門整成三個部門，核心不是集中編制，而是重建能力歸屬。前台交付回到數位發展部，資料平台與智慧分析回到數據暨人工智慧發展部，中台與工程維持在中台資訊部。

單位重組方式

  • 數位發展部 + 行銷資訊部之前台交付單位 → 數位發展部（擴編）
  • 數據暨人工智慧發展部 + 行銷資訊部之資料平台單位 → 數據暨人工智慧發展部（擴編）
  • 中台資訊部 + 台中軟體開發科 → 中台資訊部（擴編）
  • 原行銷資訊部職能重組後，分別併入三個新部門

優點

  • 前台經營與前台交付合一，需求形成到系統上線的責任鏈最清楚。
  • 智慧分析、資料治理、數據工程形成完整能力鏈，不會被單一通路專案吞沒。
  • 中台工程保留共用性，較符合壽險核心服務與平台工程的長期定位。

主要風險

  • 前台交付與資料平台重組，需要較高的轉編設計與治理重畫成本。
  • 台中軟體開發科與前台系統科之分工，需要在轉編前定義清楚。
  • 若需求排序機制不足，數位發展部仍可能出現產品端壓過交付端的情況。

  • 數位發展部 + 行銷資訊部之前台交付單位 + 台中軟體開發科 → 數位發展部（擴編）
  • 數據暨人工智慧發展部 + 行銷資訊部之資料平台單位 → 數據暨人工智慧發展部（擴編）
  • 中台資訊部維持原四個單位 → 中台資訊部（維持）
  • 原行銷資訊部職能重組後，分別併入三個新部門

  • 台中軟體開發科若長期只服務前台，工程能量可能逐步產品化，弱化共用能力。
  • 數位發展部主管幅度明顯放大，管理難度高於方案 A。
  • 前台短期效率較高，但中長期容易形成各自開發、共用平台不足的情況。

推進路徑

建議的分階段做法

推進順序的目的，是降低一次性整併的執行風險，並把組織調整與治理設計同步完成。若只做組織圖調整、不做治理補強，整併效果通常無法落地。

第一階段：先分能力

先確認前台交付、資料平台、智慧分析、中台工程的單位歸屬與需求排序機制，避免先宣布整併、後面再回頭修邊界。

第二階段：再做轉編

依主案或備選案進行單位轉編、權責重畫與主管鏈調整，同步處理 KPI、預算與專案治理，不只換組織圖。

第三階段：補治理

整併完成後，建立需求管理委員會、資料治理委員會與平台架構審議機制，否則問題會從部門間協調，變成部門內排擠。

不建議組合

不建議的五種組合與主要風險

下列做法的共同問題，不在名稱，而在於會同時破壞能力邊界、責任鏈與治理節奏。即使短期看似能壓低部門數，中期仍會把協調成本與治理風險拉高。

治理風險

資料治理、需求治理與架構治理責任不清。

營運風險

前台需求、平台投資與交付優先順序互相排擠。

組織風險

主管幅度過大，協作成本轉為部門內衝突。

技術風險

平台共用性下降，重複開發與架構分裂增加。

轉型風險

整併過程無法落地，最終只剩名稱調整。

高風險

1. 直接把數位發展部與數據暨人工智慧發展部合成一個部門

看起來像「數位加數據」，但實際上是把通路經營、會員經營、前台產品、資料治理、數據工程、智慧分析全部放進同一主管鏈。前台需求的時效性與資料治理的制度性天然衝突，最後通常是資料治理、資料品質與數據工程被迫讓位。

典型後果：客戶智能與商業智能被短期專案切碎、資料治理推不動、資料平台投資長期不足。

高風險

2. 將行銷資訊部整體併入數位發展部

這種做法雖然能快速把前台相關部門做大，但會把前台經營、前台交付、資料平台與部分工程能力同時放入數位發展部。結果不是端到端變清楚，而是不同節奏的能力被迫落在同一主管鏈。

典型後果：前台效率短期上升，但資料平台投資、分析環境治理與工程共用能力會被持續擠壓。

中高風險

3. 將前台交付與資料平台全部留在行銷資訊部

前台交付與資料平台雖然都在行銷資訊部內，但兩者的服務對象、投資節奏與治理責任並不相同。若整體留在同一部門，看似省事，實際上會延續現有斷點，讓資料平台無法成為全公司可共用的能力底座。

典型後果：資料平台被視為特定前台應用的支援功能，而非企業級資料資產與模型服務能力。

中高風險

4. 把資料平台留在行銷資訊部，把分析應用留在數據暨人工智慧發展部

資料平台、數據架構、資料治理與分析應用若分屬兩個部門，短期看似平衡，長期會形成「平台的人不知道應用優先順序，應用的人無法主導平台投資」的結構性問題。

典型後果：平台建設與分析應用脫節，資料品質責任與平台責任互相推諉。

中高風險

5. 讓中台資訊部直接轉型為前台交付部門

業務中台、元件組裝、技術中台的職掌重心是共用性、壽險核心服務與技術平台。若直接改造成前台交付部門，短期雖可補人力，長期卻會傷害共用元件與平台治理的穩定性。

典型後果：每個前台專案都能拿到中台資源，但中台能力無法持續積累，重複開發增加。

長期穩定性3/5

適用情境：短期必須維持行銷資訊部完整編制，但仍需先把四部門降為三部門。

結論

建議採行順序

  • 第一順位：方案 A。它最能同時滿足部門數下降、能力邊界清楚、資料治理完整與中台共用性保留四個目標。
  • 第二順位：方案 B。適合短中期強調前台速度的情境，但須接受數位發展部規模放大與工程共用性下降的代價。
  • 第三順位：方案 C。可作為短期過渡，但不宜作為最終型態，因為前台交付與中台工程仍混在同一部門。
補充判斷

關於「數位 + 數據」是否應直接合併

  • 數位發展部的核心是通路經營、客戶經營與前台產品。
  • 數據暨人工智慧發展部的核心是智慧分析、資料治理與數據工程。
  • 兩者可以高度協作，但不宜直接合成單一部門。
  • 較合理的做法，是讓兩者在同一高階主管視角下協同，而不是在同一部門內彼此競爭資源。
