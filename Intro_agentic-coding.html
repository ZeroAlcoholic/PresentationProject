企業 AI 部門技術轉型方法論

從傳統 Coding
到 Spec-Driven Agentic Coding

現代開發的價值不在「把 code 寫快一點」，
而在「把公司問題變成可驗收的技術交付」。

核心論點：Agentic Coding 不是讓 AI 更快幫人寫 code， 而是把企業問題轉成 AI agent 可理解、可執行、可驗收的技術工作包。 SDD 是這套模式的核心方法，負責把模糊需求轉成 spec → plan → tasks → verification 的工程閉環。
傳統 Coding：工程師接到的是被切好的 coding slices
業務痛點→ BA / PM 梳理→ 架構規劃→ 工程師接 slices→ QA / UAT→ 回修迴圈
Spec-Driven Agentic：從痛點到交付的可迭代閉環
痛點釐清→ SDD Spec→ Plan / Tasks→ Agent Build→ 企業驗收→ 交付 ✓
為何現在

瓶頸已經移位了

傳統軟體的生產瓶頸是「誰來寫」。Agentic Coding 讓瓶頸轉移到「誰能把問題拆清楚、規格定準、驗收設好」。

「AI 是 放大器（amplifier）——它會放大強流程，也會放大弱流程。越早練會問題拆解、規格撰寫與任務切分，越早把 AI 從局部工具升級成部門交付能力。」

— DORA 2025 State of DevOps Report

傳統 Coding 的瓶頸

人力不足：工程師人數決定產出上限
技術債累積：每次改動需重理解歷史脈絡
交付週期長：需求 → 上線平均數週
測試覆蓋低：人工測試耗時，覆蓋率難提升
回修迴圈：QA/UAT → ticket → re-code → 再驗收

Agentic Coding 的新瓶頸

問題定義模糊：spec 不清，AI 會猜，猜錯快速放大
驗收標準缺失：「完成」定義不明，無法驗收
Non-goals 未定：範疇邊界模糊，agent 越界
repo context 不完整：agent 需理解系統才能正確操作
無迭代框架：缺少 spec → task → 驗收的閉環機制
💡
核心論點

要改變的不是「誰寫 code」，而是「誰能把問題變成可交付的技術工作包」。AI PM、AI 資料科學家與 AI 工程師的能力邊界都在上移，不是消失。

SDD 方法

SDD 是 Agentic Coding 的控制面

沒有 SDD，AI 會猜。有 SDD，AI 有明確的問題、範圍、任務邊界與驗收標準。SDD 不是多寫文件——它是把模糊需求轉成可執行、可驗收、可重跑的技術工作包。

中間不是單次通關，而是三層迭代

1
規格迭代

Spec 不清楚就修改 Spec，而不是重寫 code。良好的 spec 是後續所有迭代的基礎。

↻
2
任務迭代

測試不過就調整任務拆解，而不是放棄驗收。每個任務都要有獨立的驗收條件。

↻
3
驗收迭代

UAT 不過就回到 Spec，而不是繞過驗收。好的 Agentic Flow 不怕多次迭代。

良好的 Agentic Flow 不怕多次迭代——它怕的是「沒有規格、沒有任務邊界、沒有驗收證據」的反覆試錯

三個核心操作技巧（實際怎麼做）

TECHNIQUE 01
三段式 Prompt 結構
不要一次給一個大 Prompt——分 Discovery / Planning / Execution 三段確認，每段完成後才繼續
Stage 1：「請探索 repo，了解架構與測試框架，不修改任何檔案。完成後列出 3 個疑問再等確認。」
Stage 2：「根據以下 Spec 列出實作計畫。只列計畫，不實作，等我確認後再開始。」
Stage 3：「執行 T1。完成後自行跑驗收，測試不過請修正後重跑，直到通過。」
一次一個段落確認，避免 AI 在未理解完整情境前越界操作，也讓人有機會在每個節點調整方向
TECHNIQUE 02
禁區設定（比目標更重要）
在 Spec 的 Non-goals 明確列出：「不包含 X、不修改 Y、不碰 Z」
在 CLAUDE.md 寫入全域禁止規則，讓 agent 在整個 repo 都知道不能碰什麼
在每個 Prompt 重申當前邊界：「只做 T1，不要主動修改其他模組」
Non-goals 範例：「不處理掃描圖片 PDF（手寫辨識排第二版）、不做 UI 重設計、不支援多物流商」
AI 預設行為是「幫你多做一點」。沒有明確禁區，它會越界越改越多——而且你通常不會馬上發現
TECHNIQUE 03
驗收優先設計
在開始前就寫好 Acceptance Criteria 與 Verify 指令，不等 AI 完成再想「要怎麼驗」
準備好測試資料（測試集、邊界條件樣本），讓驗收可以自動執行不依賴人眼
在 Prompt 明確要求：「完成後自己跑驗收指令，測試不過請分析原因並修正後重跑」
Verify by：npm run test:pdf-accuracy -- --threshold=0.95 && npx playwright test pdf-flow/
驗收標準是 AI 判斷「完成」的唯一依據。不事先定義，AI 會自我判斷「差不多了」——人也沒有客觀標準可以拒絕

落地核心

從 prototype 到 production，補上的不是更多 code，而是六個可量化的驗收面向。能把問題定清楚、驗收設好的人，是 Agentic 時代最有槓桿的角色——無論是 PM、資料科學家還是工程師。

導入路線

五步建立新型態工作模式

轉型不是一次性的工具學習，而是逐步建立的工作習慣。從第一個試點開始，讓每個步驟都有可驗收的成果。

1
校準認知，對齊團隊

這不是「讓 AI 幫你寫更多 code」，而是「讓部門能接住更大範圍的業務問題」。在開始前，讓所有相關人員對新工作方式有共識：誰負責定義問題、誰設計驗收標準、AI 在哪個環節介入、人在哪個環節把關。對齊步驟跳過，是導入失敗最常見的原因。

工作方式對齊先行角色分工說清楚人定義，AI 執行
2
選擇試點任務：從容易量化的開始

第一個試點最重要的任務是「讓流程跑通一次」，而不是選最複雜的問題。選符合條件的工作：輸入 / 輸出清楚、有測試資料、非安全關鍵、改壞了容易回滾。報表、文件處理、資料清洗是最適合的起點。

清楚 I/O有測試資料低風險可回滾
3
先規劃，先定義驗收

動手之前，先把問題寫清楚（SDD Spec），再定義「什麼算完成」（Acceptance Criteria）。Non-goals 比 Goals 更難寫，也更重要——它決定了範疇邊界，讓 AI 不會越界。Verification 指令要在動手前就設計好，測試資料也要事先準備。沒有 Spec、沒有 AC 就開始，是 AI 輸出偏離、大量重做的根本原因。

Non-goals 是護欄AC 要可量化先準備測試資料
4
執行 + 把關風險

建立 CLAUDE.md / AGENTS.md（架構、慣例、禁止事項），再把 spec 拆成 3–7 個可獨立執行的任務。執行時，告訴 AI「不能做什麼」比告訴它「要做什麼」更重要——禁區設定是最容易被跳過的步驟。AI 輸出要當作草稿：關鍵數據人工抽驗、關鍵業務邏輯第二人確認、灰色判斷保留人工把關。

CLAUDE.md 必備明確禁止事項交叉驗證關鍵輸出
5
驗收交付 → 複製流程 → 部門服務化

通過六大企業驗收面向後，試點完成。接下來不是馬上擴大規模，而是把這次的 spec 格式、task 拆解、驗收清單整理成可複製的模板。讓第二個問題的導入成本大幅降低。當三到五個問題跑過這個流程，團隊就建立了真正的 Agentic 交付能力——不只是個別工具使用者，而是具備整合能力的 AI 工程部門。

流程模板化第二個問題更快部門能力而非個人技能

新型態工作模式的核心習慣只有四個：先對齊、先規劃、訂立驗收、把關風險。這四個習慣決定了 AI 最終能幫到你多少。前三個月先求一個試點跑通，建立信心與可複製的流程，再逐步擴大。

✅ Verification Checklist 模板
## Verification Checklist

Feature: [功能名稱]
Date:    [YYYY-MM-DD]

### 自動化驗收
- [ ] npm test               — 全部通過
- [ ] npm run lint           — 無錯誤
- [ ] npx tsc --noEmit      — 無型別錯誤
- [ ] npm run build          — 建置成功
- [ ] CI pipeline            — 通過（如有）

### 六大企業驗收面向
- [ ] 功能正確：主要流程 + 邊界條件 + 錯誤路徑
- [ ] 資料可信：準確率 ≥ 閾值，異常率 < 上限
- [ ] AI 評估：evaluation harness 通過，結果可重現
- [ ] 系統品質：效能、穩定性、錯誤重試、監控
- [ ] 安全制度：資安掃描、法遵確認、存取控制
- [ ] 交接可運作：文件、監控告警、維運 SOP

### Spec AC 確認
- [ ] [來自 spec 的 AC 1]
- [ ] [來自 spec 的 AC 2]

### 交付確認
- [ ] Code review 完成
- [ ] 文件更新（CHANGELOG / API doc）
- [ ] CLAUDE.md 更新（如有新規則）
- [ ] 相關人員已通知
🤖 Agent-ready Prompt 模板
## Discovery Prompt（探索，不修改任何檔案）

請探索這個 repo 並回答以下問題，不要修改任何檔案：
1. 相關功能目前在哪些檔案中實作？
2. 使用的測試框架和執行指令是什麼？
3. 有哪些現有的資料模型或 schema 與此任務相關？

請先讀取 CLAUDE.md 了解系統架構，再回答上述問題。

---

## Planning Prompt（制定計畫，不實作）

根據以下 spec，為我制定實作計畫：

[貼上 SDD Feature Spec]

請列出：
1. 需要修改或新增的檔案清單
2. 建議的實作順序與理由
3. 需要注意的風險點

不要開始實作，只需提供計畫供我確認。

---

## Task Execution Prompt（執行任務）

請執行以下任務：
Task:  [任務名稱]
Goal:  [任務目標]
Files: [相關檔案]

Steps:
  1. [步驟 1]
  2. [步驟 2]

Verification: npm test [測試路徑]

完成後請跑 Verification 指令並回報結果。
測試失敗請分析原因並修正後重跑，不要放棄驗收。
